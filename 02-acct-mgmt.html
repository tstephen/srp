1. Account Management

1.1. Roles

Roles are what grant users permission to access the system and add, change or delete data. A summary of the roles that exist and the permissions they confer is shown below:

  • User - The initial state of a user after registration. No specific capabilities beyond viewing own profile page.

  • Contributor - Able to view, edit and export data of the single organisation (s)he belongs to ;

  • Approver - All functionality of the Contributor role and additionally the ability to submit and restate annual data ;

  • Analyst - Ability to:

    • View and export but not edit data from all organisations. This is expected to be reserved to members of the SDU though there may be need for others to have it too ;

    • View and edit reference data such as Carbon factors.

  • Administrator - Initially at least one user within the SDU and one from ourselves. Users with this role will be able to:

    • View, edit and export all data for all organisations ;

    • Assign and revoke roles of other users ;

    • View and process Support Tickets.

1.2. Register User

1.2.1. Actor

Prospective user him or herself

1.2.2. User story

The user comes to the site and is presented with the login dialogue. Not yet having an account the user may click a link to register a new account. In this form the user provides an email address, organisation name, agrees terms amnd conditions and submits the form.

The system shows a confirmation message and re-presents the login dialogue.

1.2.3. User outcome

The user receives an email with an initial password included and may login with it. Once logged in the user may change the password if desired.

If the user is the first from a given organisation (s)he will automatically become the approver role. In this case the users profile page and application home page will contain links to register an organisation.

Otherwise a request will be sent to the current Approver to grant (or deny) the necessary roles. In the latter case the user will not be able to access anything beyond his or her own profile page until approved.

1.2.4. Status

In progress

1.3. Register new organisation

1.3.1. Actor

Staff of healthcare organisation (e.g. Trust, Care home etc.)

1.3.2. User story

The user needs to deliver the reporting requirements of his or her organisation and has become aware of these tools from the SDU newsletter, a colleague etc. but no one in the organisation has yet created an account as far as can be ascertained.

The user comes to the site and registers an account using Register user. Next (s)he can supply some basic organisational details. The system will then check the supplied details are unique and if so assign the user the ownership of that organisation approver role. Alternately if the organisation is already registered both the user and the existing approver will be notified in order that the approver can join the user to the organisation.

Should this process fail for any reason such as the original approver has left the organisation without passing on ownership the user may raise a support ticket on the site, which will be handled manually.

1.3.3. User outcome

The user should be able to access the system on behalf of his or her organisation.

1.3.4. Pre-requisites

Register user

1.4. Reset Own Password

1.4.1. Actor

Any user

1.4.2. User story

From time to time a user may forget his or her password. To allow this to be reset the user must be able to supply the registered email address. The system will then send an email with a new password.

Once back in to the system the user may change the password again if desired but minimum length and complexity rules apply.

If for any reason the user is not able to recall or access the correct email address then at this stage the only solution is to register a new account. If, worse still, the lost account belongs to the Approver then the user must raise a support ticket and we would probably require additional, offline, verification of their identity.

1.4.3. User outcome

A new password is issued and access to the system is restored.

1.4.4. Status

Available

1.5. Raise Support Ticket

1.5.1. Actor

End user of the system, typically an employee of a reporting organisation

1.5.2. User story

Two main reasons could lead the user to need to raise a support ticket:

  • The system is behaving in an unexpected way, possibly as a result of a bug or simply because the user does not understand it in some way. Termed 'Application' problem.

  • The user has a question regarding the requested data or wishes to have it presented out in a different way. These are Data problems.

It is possible to record support tickets directly within the system including:

  • A short description of the problem

  • A screenshot of the application’s current state

  • Category of problem as above: Application, Data, Other

1.5.3. User outcome

The user receives an email confirming the ticket’s submission and the ticket is placed into the task queue of one or more groups of users, based on the category of the problem.

1.5.4. Status

In progress

1.6. Process Support Tickets

1.6.1. Actors

  • Primary: System administrator or SDU analyst

  • Secondary: End user of the system

Requires admin role.

1.6.2. User story

When new support tickets are raised they are placed in a queue that records when they are opened, closed and any activities in between if it proves impossible to resolve straight away.

The user can see this list in a summary 'inbox' and choose which one (s)he wishes to open. From there the following actions exist:

  • Defer the ticket for a specific period of time

  • Request further information of the end user

  • Add notes

  • Close the ticket

In all cases the system will send a notification to the end user if public information has been provided. It is only possible to close the ticket with a public comment to explain to the end user.

1.6.3. User outcome

The ticket will typically disappear from the inbox, either because further information has been requested or because it has been deferred. However it is also possible to open and view a ticket and take no action, in which case it stays in the list.

1.6.4. Status

In progress

1.7. Reset Other’s Password

1.7.1. Actor

  • Primary: System administrator or SDU analyst

  • Secondary: End user of the system

Requires admin role.

1.7.2. User story

In normal use any user can reset his or her own password using Reset own password. However there may be circumstances in which it is simply more convenient for an administrator to reset the password on the user’s behalf.

The administrator accesses a full list of all users and from there initiates an email to send the user a new password. Note that the administrator cannot see the old or the new passwords.

1.7.3. Exclusions

This story does not cater for the situation that the user has lost access to the registered email account as well as the password. That would pose a risk of the administrator being tricked into re-assigning an account to someone entirely different.

1.7.4. User outcome

The user is able to login using the new password received in email and optionally to reset that password.

1.7.5. Status

Available
