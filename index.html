1. Introduction

This project provides data tools to support evidence-based policies to reduce social and environmental impacts of the health system. It is a collaboration between the Sustainable Development Unit, private industry and individuals within the NHS responsible for these policies and their delivery.

1.1. Aims

  • To streamline and digitise existing reporting efforts of the SDU.

  • To encourage collaboration on tools and break down silo-ed efforts.

  • To foster a community of the interested and responsible for these issues.

This documentation serves two different but overlapping audiences:

  • Analysts and Reporters of data relating to all kinds of non-financial, non-clinical impacts of the NHS. This material is structured around a set of user stories so you can quickly find the way that the toolkit is able to satisfy problems you already recognise.

  • Technologists interested in maintaining, extending or using the data tools. For these people we have API documentation and an overview of the software architecture.

1.2. Licensing

  • Software is licensed under the Apache license v2.

  • Data is the property of the reporting organisation until and unless they choose to submit it to the SDU. The SDU will release it publically under the Open Government Licence in the December following the reporting cycle ending March.

We support the NHS, public health and social care to embed and promote the three elements of sustainable development - environmental, social and financial.

The Unit is jointly funded by, and accountable to, NHS England and Public Health England to ensure that the health and care system fulfils its potential as a leading sustainable and low carbon service.

sduhealth.org.uk

2. Account Management

2.1. Roles

Roles are what grant users permission to access the system and add, change or delete data. A summary of the roles that exist and the permissions they confer is shown below:

  • User - The initial state of a user after registration. No specific capabilities beyond viewing own profile page.

  • Contributor - Able to view, edit and export data of the single organisation (s)he belongs to ;

  • Approver - All functionality of the Contributor role and additionally the ability to submit and restate annual data ;

  • Analyst - Ability to:

    • View and export but not edit data from all organisations. This is expected to be reserved to members of the SDU though there may be need for others to have it too ;

    • View and edit reference data such as Carbon factors.

  • Administrator - Initially at least one user within the SDU and one from ourselves. Users with this role will be able to:

    • View, edit and export all data for all organisations ;

    • Assign and revoke roles of other users ;

    • View and process Support Tickets.

2.2. Register User

2.2.1. Actor

Prospective user him or herself

2.2.2. User story

The user comes to the site and is presented with the login dialogue. Not yet having an account the user may click a link to register a new account. In this form the user provides an email address, organisation name, agrees terms amnd conditions and submits the form.

The system shows a confirmation message and re-presents the login dialogue.

2.2.3. User outcome

The user receives an email with an initial password included and may login with it. Once logged in the user may change the password if desired.

If the user is the first from a given organisation (s)he will automatically become the approver role. In this case the users profile page and application home page will contain links to register an organisation.

Otherwise a request will be sent to the current Approver to grant (or deny) the necessary roles. In the latter case the user will not be able to access anything beyond his or her own profile page until approved.

2.2.4. Status

In progress

2.3. Register new organisation

2.3.1. Actor

Staff of healthcare organisation (e.g. Trust, Care home etc.)

2.3.2. User story

The user needs to deliver the reporting requirements of his or her organisation and has become aware of these tools from the SDU newsletter, a colleague etc. but no one in the organisation has yet created an account as far as can be ascertained.

The user comes to the site and registers an account using Register user. Next (s)he can supply some basic organisational details. The system will then check the supplied details are unique and if so assign the user the ownership of that organisation approver role. Alternately if the organisation is already registered both the user and the existing approver will be notified in order that the approver can join the user to the organisation.

Should this process fail for any reason such as the original approver has left the organisation without passing on ownership the user may raise a support ticket on the site, which will be handled manually.

2.3.3. User outcome

The user should be able to access the system on behalf of his or her organisation.

2.3.4. Pre-requisites

Register user

2.4. Reset Own Password

2.4.1. Actor

Any user

2.4.2. User story

From time to time a user may forget his or her password. To allow this to be reset the user must be able to supply the registered email address. The system will then send an email with a new password.

Once back in to the system the user may change the password again if desired but minimum length and complexity rules apply.

If for any reason the user is not able to recall or access the correct email address then at this stage the only solution is to register a new account. If, worse still, the lost account belongs to the Approver then the user must raise a support ticket and we would probably require additional, offline, verification of their identity.

2.4.3. User outcome

A new password is issued and access to the system is restored.

2.4.4. Status

Available

2.5. Raise Support Ticket

2.5.1. Actor

End user of the system, typically an employee of a reporting organisation

2.5.2. User story

Two main reasons could lead the user to need to raise a support ticket:

  • The system is behaving in an unexpected way, possibly as a result of a bug or simply because the user does not understand it in some way. Termed 'Application' problem.

  • The user has a question regarding the requested data or wishes to have it presented out in a different way. These are Data problems.

It is possible to record support tickets directly within the system including:

  • A short description of the problem

  • A screenshot of the application’s current state

  • Category of problem as above: Application, Data, Other

2.5.3. User outcome

The user receives an email confirming the ticket’s submission and the ticket is placed into the task queue of one or more groups of users, based on the category of the problem.

2.5.4. Status

In progress

2.6. Process Support Tickets

2.6.1. Actors

  • Primary: System administrator or SDU analyst

  • Secondary: End user of the system

Requires admin role.

2.6.2. User story

When new support tickets are raised they are placed in a queue that records when they are opened, closed and any activities in between if it proves impossible to resolve straight away.

The user can see this list in a summary 'inbox' and choose which one (s)he wishes to open. From there the following actions exist:

  • Defer the ticket for a specific period of time

  • Request further information of the end user

  • Add notes

  • Close the ticket

In all cases the system will send a notification to the end user if public information has been provided. It is only possible to close the ticket with a public comment to explain to the end user.

2.6.3. User outcome

The ticket will typically disappear from the inbox, either because further information has been requested or because it has been deferred. However it is also possible to open and view a ticket and take no action, in which case it stays in the list.

2.6.4. Status

In progress

2.7. Reset Other’s Password

2.7.1. Actor

  • Primary: System administrator or SDU analyst

  • Secondary: End user of the system

Requires admin role.

2.7.2. User story

In normal use any user can reset his or her own password using Reset own password. However there may be circumstances in which it is simply more convenient for an administrator to reset the password on the user’s behalf.

The administrator accesses a full list of all users and from there initiates an email to send the user a new password. Note that the administrator cannot see the old or the new passwords.

2.7.3. Exclusions

This story does not cater for the situation that the user has lost access to the registered email account as well as the password. That would pose a risk of the administrator being tricked into re-assigning an account to someone entirely different.

2.7.4. User outcome

The user is able to login using the new password received in email and optionally to reset that password.

2.7.5. Status

Available

3. Non-Financial Reporting

3.1. Enter Report Data

3.1.1. Actor

End user of the system, typically an employee of a reporting organisation

3.1.2. User story

The user will be able to enter the same data as currently requested in the spreadsheet. Exact nature and structure of this data is described in the data model.

The user may enter one year’s data in a single page. All entries are automatically saved minimising the risk of any being lost. It is possible to scroll to previous and subsequent years, providing they exist. Only one year into the future will be available.

None of this data is shared with the SDU until it is submitted by a suitably qualified user within the reporting organisation. Suitably qualified means a user with the Approver role.

3.1.3. User outcome

The user may login and change any data (s)he wishes until prior to the report being submitted. After that it may only be viewed, or copied to created a revised submission. See here for details of that.

3.1.4. Status

Visual design

3.2. Submit Report

3.2.1. Actor

Reporting organisation user with approver role.

3.2.2. Pre-requisites

  • Register user

  • Register organisation

  • Enter report data

3.2.3. User story

Once the Approver has satisfied him or herself that the provided data is accurate (s)he may submit it to the SDU. This will include a reminder of the terms and conditions presented at registration.

3.2.4. User outcome

Data is submitted to the SDU, and becomes read-only to the reporting organisation. Subsequent changes may be made by re-stating the submission.

3.2.5. Status

Visual design

3.3. Submit Revised Report

3.3.1. Actor

Reporting organisation user with approver role.

3.3.2. Pre-requisites

  • Submit Report

3.3.3. User story

If, after submission, revision is necessary it will be possible to create a complete copy of a given year’s return and submit that. This submission will be clearly labelled as being a restated submission.

3.3.4. User outcome

All 'live' reports will be replaced without any further user interaction. Obviously those created for offline use, for example as PDFs, will need to be re-created.

3.7. Maintain Reference Data

3.7.1. Actor

SDU analyst or System administrator

Requires admin role.

3.7.2. User story

Several pieces of data are expected to change over time and will dramatically affect the calculations reported. For example the Carbon intensity of certain operations will change as the wider economy is gradually de-carbonised.

Once logged in the user will the available application pages including any maintenance pages. Navigating to the appropriate one, the user will be able to enter or update the appropriate values.

Most, perhaps all, reference data will have an applicability period so there should not be any risk of confusion due to reports changing after being submitted so long as reference data is defined ahead of the reporting period. Should retrospective changes need to be made appropriate communication to users should accompany it.

3.7.3. Exclusions

Note that all reference data will need to have values provided as part of the initial system implementation or the calculations will not work. Therefore it is not necessary to add new reference data but only to update the existing values.

3.7.4. User outcome

Reports should reflect the changed reference data on next viewing.

5. Data Model

6. Architecture

The system is built 'API-first'. In other words the data is hosted on a cloud server and a series of services exposed to permit access to it. These services represent the Application Programming Interface (API) and support not only the web application at https://srp.digital but potentially other applications that may be built in the future by the community.

The front-end is built as a 'Progressive Web Application' which means that users need only a reasonably recent browser to access the system. The progressive part is that certain features, such as offline access, may only be available to the most modern browsers but a graceful fallback should exist back to at least Internet Explorer 10.

On the server side, the application is built on Spring Boot with a MySQL data store.

7. For Developers

The architecture of the system is described here. This page gives the basic 5 minute guide to get the software installed and running on your machine.

7.1. Pre-requisites

Before starting please ensure you have the following basic tools installed:

  • Java 8: Get and install the JDK for your operating system from Oracle site. Version 8 is the minimum version required

  • Maven is used to build the Java source code

  • (optional) MySQL: There is an embedded database used by default but this is only suitable for demonstration purposes. Even in development you will probably want to switch this for a more robust database. The system is developed using MySQL though there should be minimal effort to use others such as PostgreSQL.

7.2. Source Code

Source code is held and versioned on Github, more information may be found at TODO.

7.3. Building

Once you have Maven installed and the source code checked out it, it is a simple matter of running this command to build all the artefacts.

You may compile and run the server locally using an embedded database with some minimal test data using this command:

  mvn clean spring-boot:run

Having done so open your browser with this address:

  http://localhost:8090/

Spring Boot takes an opinionated view of building applications, in other words there are defaults for everything so that it works with minimal code but you gradually remove these defaults and replace them with your own if you wish as requirements grow and change. There is lots of good documentation and sample projects that you can read about here.
