1. Non-Financial Reporting

1.1. Enter Report Data

1.1.1. Actor

End user of the system, typically an employee of a reporting organisation

1.1.2. User story

The user will be able to enter the same data as currently requested in the spreadsheet. Exact nature and structure of this data is described in the data model.

The user may enter one year’s data in a single page. All entries are automatically saved minimising the risk of any being lost. It is possible to scroll to previous and subsequent years, providing they exist. Only one year into the future will be available.

None of this data is shared with the SDU until it is submitted by a suitably qualified user within the reporting organisation. Suitably qualified means a user with the Approver role.

1.1.3. User outcome

The user may login and change any data (s)he wishes until prior to the report being submitted. After that it may only be viewed, or copied to created a revised submission. See here for details of that.

1.1.4. Status

Visual design

1.2. Submit Report

1.2.1. Actor

Reporting organisation user with approver role.

1.2.2. Pre-requisites

  • Register user

  • Register organisation

  • Enter report data

1.2.3. User story

Once the Approver has satisfied him or herself that the provided data is accurate (s)he may submit it to the SDU. This will include a reminder of the terms and conditions presented at registration.

1.2.4. User outcome

Data is submitted to the SDU, and becomes read-only to the reporting organisation. Subsequent changes may be made by re-stating the submission.

1.2.5. Status

Visual design

1.3. Submit Revised Report

1.3.1. Actor

Reporting organisation user with approver role.

1.3.2. Pre-requisites

  • Submit Report

1.3.3. User story

If, after submission, revision is necessary it will be possible to create a complete copy of a given year’s return and submit that. This submission will be clearly labelled as being a restated submission.

1.3.4. User outcome

All 'live' reports will be replaced without any further user interaction. Obviously those created for offline use, for example as PDFs, will need to be re-created.

Visual design

1.7. Maintain Reference Data

1.7.1. Actor

SDU analyst or System administrator

Requires admin role.

1.7.2. User story

Several pieces of data are expected to change over time and will dramatically affect the calculations reported. For example the Carbon intensity of certain operations will change as the wider economy is gradually de-carbonised.

Once logged in the user will the available application pages including any maintenance pages. Navigating to the appropriate one, the user will be able to enter or update the appropriate values.

Most, perhaps all, reference data will have an applicability period so there should not be any risk of confusion due to reports changing after being submitted so long as reference data is defined ahead of the reporting period. Should retrospective changes need to be made appropriate communication to users should accompany it.

1.7.3. Exclusions

Note that all reference data will need to have values provided as part of the initial system implementation or the calculations will not work. Therefore it is not necessary to add new reference data but only to update the existing values.

1.7.4. User outcome

Reports should reflect the changed reference data on next viewing.
